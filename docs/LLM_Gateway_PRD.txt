1. Executive Summary
This document defines the complete product requirements for the LLM Gateway — a production-grade, multi-provider AI routing infrastructure. The gateway functions as a unified control plane sitting between all internal services and external LLM providers (OpenAI, Anthropic, AWS Bedrock, Google Vertex AI, Azure OpenAI, Mistral, Groq, Cohere, and self-hosted vLLM clusters).

The architecture makes a deliberate language split: the performance-critical hot path — request normalization, semantic caching, guardrails, routing, and response streaming — is written in Rust for zero-cost abstractions and sub-millisecond P99 overhead. Supporting services — the admin API, config management, provider health monitoring, audit pipeline, and observability collector — are written in Go for development velocity and ecosystem breadth.

NOTE
This is not a demo proxy. The target is: <1 ms added P99 latency at 10,000 RPS, 99.99% availability, zero-downtime deployments, and full cost attribution across every team, workspace, and model in the organisation.

1.1 Business Objectives
	•	Vendor independence: eliminate hard dependencies on any single LLM provider. Provider migrations must require zero application-layer changes.
	•	Cost governance: give finance and engineering leadership real-time visibility into token spend per team, per model, and per product feature. Enable hard budget enforcement.
	•	Reliability: shield applications from provider outages via automatic failover. Provider P99 tail latencies must not propagate to end users.
	•	Security and compliance: enforce PII redaction, prompt injection detection, and audit logging on every request. Required for SOC 2 Type II and ISO 27001.
	•	Developer experience: internal teams call a single OpenAI-compatible endpoint. They never manage provider credentials or implement retry logic.

3.2 Component Topology
NOTE
Deployment topology: Clients → L4 Load Balancer → Gateway Pods (Rust binary, 3+ replicas, stateless) → [Redis, Qdrant, Postgres] + [LLM Providers]. All shared state lives in external stores. Gateway pods are cattle, not pets.

3.3 Data Flow — Happy Path
	•	Client sends POST /v1/chat/completions with a virtual API key.
	•	Gateway validates the JWT / virtual key (Rust, in-memory cache of key metadata).
	•	Rate-limit check against Redis sliding window (Rust + redis-rs).
	•	Pre-call guardrail pipeline: PII scan → injection detection → topic guard.
	•	Semantic cache lookup: embed prompt with in-process ONNX model → ANN query Qdrant.
	•	Cache hit: return cached response with Cache-Status: HIT header. End.
	•	Cache miss: intelligent router selects provider (circuit state, latency score, cost weight).
	•	Request enrichment: inject trace ID, workspace metadata, model normalisation.
	•	Forward to provider. Stream response back chunk-by-chunk (SSE passthrough).
	•	Post-call guardrail scan on completion tokens.
	•	Async: emit audit event to Kafka → ClickHouse; update token counters in Redis.
	•	Store embedding + response in Qdrant (semantic cache write).
	•	Return response to client.

4. Functional Requirements
4.1 Request Proxy Engine  
RUST COMPONENT
Implemented in Rust using Tokio async runtime, Hyper HTTP server, and reqwest for upstream calls.

4.1.1 OpenAI-Compatible API Surface
	•	FR-101: The gateway MUST expose POST /v1/chat/completions, POST /v1/completions, POST /v1/embeddings, and POST /v1/images/generations with request/response schemas 100% compatible with the OpenAI API specification v1.
	•	FR-102: The gateway MUST support both streaming (stream: true, SSE) and non-streaming modes. Streaming MUST forward chunks to the client with zero buffering of the full response.
	•	FR-103: The gateway MUST normalise provider-specific request and response formats via adapter modules. Clients always receive OpenAI-schema responses regardless of the underlying provider.
	•	FR-104: The gateway MUST inject a unique request ID (X-Request-ID header) on every request and propagate it to the provider call and all log/trace records.
	•	FR-105: The gateway MUST support multi-modal inputs (text + images) where the target provider supports them, passing through base64-encoded image content without modification.

4.1.2 Provider Adapters
Each adapter implements a Rust trait:
trait ProviderAdapter: Send + Sync { fn normalize_request(...) -> ProviderRequest; fn normalize_response(...) -> OpenAIResponse; fn stream_chunk(...) -> Option<StreamChunk>; }
	•	FR-106: Adapters MUST be implemented for: OpenAI, Anthropic Claude, AWS Bedrock (Claude + Titan + Llama variants), Google Vertex AI (Gemini), Azure OpenAI, Cohere Command, Mistral AI, Groq, and a generic OpenAI-compat adapter for vLLM / Ollama self-hosted endpoints.
	•	FR-107: Adapters MUST translate stop sequences, temperature, top_p, max_tokens, and tool/function call schemas bidirectionally.
	•	FR-108: The adapter registry MUST be extensible — adding a new provider requires implementing the ProviderAdapter trait and registering it; no core modification required.

4.1.3 Connection Management
	•	FR-109: The gateway MUST maintain a persistent per-provider HTTP/2 connection pool. Pool size MUST be configurable per provider (default: 50 connections).
	•	FR-110: The gateway MUST implement a three-tier timeout: (a) time-to-first-token (TTFT) timeout, (b) total stream duration timeout, (c) provider connect timeout. All MUST be configurable per route.
	•	FR-111: The gateway MUST validate TLS certificates for all upstream provider connections. Certificate pinning MUST be supported for self-hosted endpoints.

4.2 Intelligent Router
RUST COMPONENT
Implemented in Rust. Lock-free atomic state per provider. Routing decisions in <50 µs.

4.2.1 Route Configuration
	•	FR-201: Operators MUST be able to define route groups with: a primary provider, up to 5 ordered fallback providers, per-provider weights (for load distribution), and allowed model overrides.
	•	FR-202: Route configs MUST support model aliasing — clients can call model: "production-fast" and the gateway resolves it to gpt-4o-mini or claude-haiku based on the active routing policy.
	•	FR-203: Routing decisions MUST be deterministic and auditable — the chosen provider and reason (primary / weight / failover / budget) MUST appear in every trace.

4.2.2 Circuit Breaker
	•	FR-204: Each provider MUST have an independent circuit breaker with three states: Closed (normal), Half-Open (recovery probe), and Open (failed).
	•	FR-205: Circuit open conditions (all configurable): error rate > threshold over a rolling 60-second window, or consecutive failure count > N.
	•	FR-206: In Open state, the router MUST skip the provider entirely and choose the next-best fallback. If all providers are Open, return HTTP 503 with Retry-After.
	•	FR-207: In Half-Open state, the router MUST route 5% of traffic to probe recovery. On 2 consecutive successes, transition to Closed. On any failure, return to Open.
	•	FR-208: Circuit breaker state MUST be stored in process memory (AtomicU8) — not Redis — to avoid adding I/O on the hot path. State is per-instance (acceptable eventual consistency).

4.2.3 Load Distribution & Scoring
	•	FR-209: When multiple providers are healthy, the router MUST score each candidate using: current observed P95 latency (EWMA), cost per 1K output tokens (configurable), current circuit state, and explicit operator-set weight.
	•	FR-210: The gateway MUST support multi-key load distribution per provider — if a workspace has registered 5 API keys for OpenAI, requests MUST be distributed across them with per-key rate-limit tracking.

4.2.4 Retry Policy
	•	FR-211: The gateway MUST retry on: HTTP 429 (Rate Limited), HTTP 503, HTTP 524 (Timeout), and TCP connection errors. It MUST NOT retry on 4xx client errors (400, 401, 403, 404, 422).
	•	FR-212: Retries MUST use exponential backoff with full jitter. If the provider returns Retry-After, that value MUST be respected as the minimum backoff.
	•	FR-213: Max retries MUST be configurable per route (default: 3). Each retry SHOULD attempt the next provider in the fallback chain rather than the same failed provider.

4.3 Semantic Cache
RUST COMPONENT
Hot path implemented in Rust. Embedding via in-process ONNX Runtime (no remote call). Vector store: Qdrant. Cold cache metadata in Redis.

4.3.1 Cache Lookup
	•	FR-301: On every non-streaming request, the gateway MUST perform a semantic cache lookup before forwarding to the provider.
	•	FR-302: The cache lookup MUST embed the canonical prompt using an in-process embedding model (all-MiniLM-L6-v2 via ONNX Runtime). The embedding call MUST complete in <5 ms at P99.
	•	FR-303: The gateway MUST query Qdrant for the top-3 nearest neighbours. If the closest neighbour has cosine similarity ≥ threshold (default: 0.93, configurable per workspace), it is a cache hit.
	•	FR-304: Cache hits MUST return the stored response with headers: X-Cache: HIT, X-Cache-Score: <similarity>, X-Cache-Age: <seconds>.
	•	FR-305: Cache misses MUST proceed to provider call. After a successful response, the gateway MUST asynchronously store the embedding + response in Qdrant (non-blocking write, background task).

4.3.2 Cache Scoping & Isolation
	•	FR-306: Cache entries MUST be scoped to a (workspace_id, model_name, system_prompt_hash) namespace. Workspace A MUST never receive a cached response from Workspace B's traffic.
	•	FR-307: Each cache entry MUST carry a model_version tag. When the gateway config references a model version change, all entries for that model MUST be invalidated.
	•	FR-308: Cache entries MUST have a configurable TTL (default: 24 hours). Entries with temperature > 0.3 SHOULD have a reduced TTL (default: 1 hour) since responses are non-deterministic.

4.3.3 Cache Bypass & Invalidation
	•	FR-309: Clients MUST be able to bypass the cache by sending Cache-Control: no-cache or X-LLM-Cache-Bypass: true.
	•	FR-310: Admin API MUST support: flush entire workspace cache, flush by model, flush by tag, and flush by time range.
	•	FR-311: Cache warming: on gateway startup, the system MUST pre-populate the Qdrant index from the last 7 days of high-frequency prompt embeddings stored in ClickHouse.

4.4 Guardrails Engine
RUST COMPONENT
Implemented in Rust as a composable middleware pipeline. Each guardrail is a trait object with a uniform interface. Pipeline stages run in series; any stage can short-circuit the request.

4.4.1 Pre-Call Guardrails (Request Scanning)
	•	FR-401 — PII Detector: MUST detect and optionally redact: SSNs, credit/debit card numbers, phone numbers (international formats), email addresses, US/Indian/EU passport numbers, bank account numbers, and IP addresses. Detection via compiled Rust regex + optional NER model (configurable). On detection: log PII event, redact tokens to [REDACTED_<TYPE>], or block request (configurable per workspace policy).
	•	FR-402 — Prompt Injection Detector: MUST score incoming prompts against a maintained pattern library of jailbreak and injection techniques (DAN, "ignore all previous instructions", role confusion, etc.). Score MUST be produced using both regex matching and semantic similarity to known attack embeddings. Configurable threshold for block vs. flag-only action.
	•	FR-403 — Topic / Scope Guard: Workspace admins MUST be able to define a scope description (free text). The guardrail classifies incoming prompts against this scope using a zero-shot classification model. Off-scope requests return HTTP 403 with a configurable deflection message.
	•	FR-404 — Content Policy Filter: MUST support workspace-level allow/deny lists (exact match and glob patterns). A global policy layer applies to all workspaces and cannot be overridden by workspace admins.
	•	FR-405 — Agent Loop Detector: For requests carrying a session_id, the gateway MUST track the tool call chain depth. When depth exceeds a configurable limit (default: 20 tool calls), the gateway MUST terminate the session and return a tool_loop_limit_exceeded error.

4.4.2 Post-Call Guardrails (Response Scanning)
	•	FR-406 — PII Leak Detector: MUST apply the same PII detection pipeline to LLM completions. PII appearing in responses is treated as a leakage event, logged separately, and optionally redacted before delivery to the client.
	•	FR-407 — Toxicity Classifier: MUST run a lightweight binary classifier on completions. Responses above the toxicity threshold MUST be blocked. The classifier MUST run in-process (no remote call) and complete in <10 ms P99.
	•	FR-408 — Schema Validation: Workspaces using JSON mode MUST be able to register a JSON Schema. The gateway validates the completion against the schema. On validation failure: retry up to 2 times with a repair prompt, then return a structured_output_failure error.
	•	FR-409 — Hallucination Guard: (Phase 2) For RAG-enabled use cases, the gateway MUST support a grounding check: validate that all factual claims in the completion are traceable to the provided context documents. Unsupported claims are flagged in the response metadata.

4.4.3 Guardrail Pipeline Configuration
	•	FR-410: Each workspace MUST be able to enable/disable individual guardrail stages independently.
	•	FR-411: Guardrail execution MUST be parallelised where stages are independent (e.g., PII scan and injection detection can run concurrently).
	•	FR-412: All guardrail actions (block, redact, flag) MUST be recorded in the audit log with: stage name, action taken, match details (hashed, not plain text for PII), and latency.

4.5 Token Budgeting & Cost Governance
RUST COMPONENT
Enforced in Rust on the hot path using atomic Redis Lua scripts for consistency. Budget metadata managed by Go admin service.

4.5.1 Budget Hierarchy
	•	FR-501: Budget enforcement MUST support a four-level hierarchy: Organisation → Workspace → Project → Virtual Key. Limits at each level apply cumulatively (a workspace cannot exceed its org limit even if its own limit is higher).
	•	FR-502: Budget dimensions MUST include: tokens per minute (TPM), tokens per day, tokens per month, USD cost per day, and USD cost per month.
	•	FR-503: Budget enforcement MUST be atomic — concurrent requests MUST NOT collectively exceed a budget. Implementation: Redis INCR + Lua script (check-and-increment) executed in a single roundtrip.

4.5.2 Virtual Key System
	•	FR-504: Every client MUST authenticate using a virtual key (format: lgw_<env>_<uuid4>). Real provider API keys MUST never be exposed to application code.
	•	FR-505: Virtual keys MUST carry: allowed_models list, max_tokens_per_request, budget_policy_id, allowed_ip_cidrs, expiry timestamp, and workspace/project tags.
	•	FR-506: Virtual key resolution to provider credential MUST use a secrets manager integration (HashiCorp Vault, AWS Secrets Manager, or GCP Secret Manager). No credentials in environment variables or config files.
	•	FR-507: The gateway MUST support zero-downtime key rotation: add new key → drain old key (wait for in-flight requests) → deactivate old key. Rotation MUST be triggerable via admin API or automated on a schedule.

4.5.3 Budget Alerts & Enforcement
	•	FR-508: When a budget reaches 80% consumed, the gateway MUST emit a warning event and (optionally) notify the workspace admin via webhook.
	•	FR-509: When a budget is exhausted, the gateway MUST return HTTP 429 with X-LLM-Budget-Exhausted: true and a Retry-After timestamp indicating when the budget resets. Requests MUST be rejected, not silently truncated.
	•	FR-510: Operators MUST be able to configure graceful degradation: instead of hard-blocking on budget exhaustion, route to a cheaper fallback model (e.g. GPT-4o → GPT-4o-mini) with budget-degraded flag in response metadata.

4.6 Authentication & Access Control
RUST COMPONENT
Auth validation (JWT/key lookup) runs in the Rust hot path. Role and permission management is handled by the Go admin service.

4.6.1 Authentication Methods
	•	FR-601: MUST support virtual key authentication via Authorization: Bearer lgw_<env>_<uuid> header (primary method for application clients).
	•	FR-602: MUST support JWT authentication for admin API access. JWTs MUST be RS256-signed. Public key rotation MUST be supported via JWKS endpoint.
	•	FR-603: MUST support mTLS for service-to-service calls in zero-trust network environments (optional, configurable per route).

4.7.3 Structured Logging
	•	FR-704: All log output MUST be JSON-structured (no free-text format). Mandatory fields per line: trace_id, span_id, level, timestamp_utc, service, workspace_id, virtual_key_id (masked), model, provider, latency_ms.

4.7.4 Audit Log
	•	FR-707: Every request MUST produce an immutable audit record written to ClickHouse asynchronously (Kafka → ClickHouse consumer). Fields: request_id, timestamp, workspace_id, virtual_key_id, model, provider_used, prompt_token_count, completion_token_count, cost_usd, latency_ms, cache_hit, guardrail_actions, circuit_breaker_state, error_code.
	•	FR-708: Audit records MUST be retained for a minimum of 13 months (SOC 2 requirement). Older records are archived to cold object storage (S3/GCS), not deleted.
	•	FR-709: The admin API MUST expose a paginated audit log query endpoint with filters: time range, workspace, virtual key, model, guardrail action, and error code.

5.2 Reliability & Availability
	•	NFR-201: The gateway MUST maintain 99.99% availability (< 52.6 minutes downtime/year) measured monthly.
	•	NFR-202: Provider failover MUST complete in < 100 ms from failure detection to first retry on fallback provider.
	•	NFR-203: Zero-downtime deployments MUST be supported. Rolling update with maxSurge: 1, maxUnavailable: 0. In-flight streaming requests MUST be drained before pod termination (preStop hook, 30s grace).
	•	NFR-204: The gateway MUST continue serving traffic (using the last known config) if Postgres or Redis becomes temporarily unavailable, for up to 60 seconds. After 60 seconds, health check MUST fail and the pod removed from load balancer rotation.

5.3 Security
	•	NFR-301: All inter-service communication MUST use TLS 1.3. mTLS is required for gateway → admin service and gateway → audit Kafka.
	•	NFR-302: Provider API keys MUST be stored in a secrets manager. MUST NOT appear in: environment variables, application config files, container images, or logs.
	•	NFR-303: The gateway MUST pass OWASP API Security Top 10 assessment before production deployment.
	•	NFR-304: Dependency audit MUST run in CI. Critical CVEs MUST block deployment. cargo audit (Rust) and govulncheck (Go) MUST be included in the pipeline.
	•	NFR-305: Data residency: the gateway MUST support routing requests to region-specific provider endpoints to comply with data residency requirements (EU data stays in EU endpoints).

5.4 Scalability
	•	NFR-401: The gateway MUST scale horizontally. Adding a pod MUST produce a proportional throughput increase (linear scaling to at least 16 pods).
	•	NFR-402: HorizontalPodAutoscaler MUST scale on two signals: CPU utilisation (target 60%) and custom metric llm_gateway_requests_total rate (target 3,000 RPS/pod).
	•	NFR-403: The semantic cache (Qdrant) MUST support sharding. The ANN index MUST remain performant at 10M+ cached embeddings.
	•	NFR-404: The audit ClickHouse cluster MUST handle 50,000 audit events/second sustained ingestion without query degradation.

6.2 Data Privacy & Residency
	•	DR-101: Prompt content MUST NOT be stored anywhere in the gateway infrastructure. Only the prompt's SHA-256 hash, token count, and embedding vector may be persisted.
	•	DR-102: Completion content MUST NOT be stored except in the semantic cache (Qdrant) for cache-hit purposes. Cache entries are scoped to workspace and have configurable TTL.
	•	DR-103: Personally Identifiable Information (PII) identified by the guardrail engine MUST be redacted from all stored records before persistence.
	•	DR-104: For EU workspaces, requests MUST be routed to EU-region provider endpoints and all data stores (Redis, Qdrant, Postgres) MUST be deployed in EU regions.

10. Risks & Mitigations
Risk
Severity
Likelihood
Mitigation
Rust learning curve slows delivery
Medium
High
Time-box Phase 1 to 4 weeks; use well-known crates (Tokio, Hyper, Axum). Establish Rust code review norms early. Go remains available for non-hot-path work.
Qdrant ANN performance degrades at scale
High
Medium
Benchmark at 1M, 5M, 10M vectors in Phase 3. Evaluate pgvector as fallback. Design cache backend as a Rust trait for easy substitution.
Provider API schema drift breaks adapters
Medium
High
Contract tests run in CI against provider spec snapshots. PagerDuty alert on adapter parse error spike. Adapter updates are isolated changes.
Redis single point of failure for budgets
High
Low
Deploy Redis in cluster mode (3 primary, 3 replica). Budget counters use Redis WAIT for durability. Fallback: disable budget enforcement on Redis unavailability (log + alert).
Semantic cache returns stale or wrong responses
High
Medium
Similarity threshold tuning (A/B test at 0.93 vs 0.95). Mandatory TTL on all entries. Cache bypass header available to all clients. Monitor cache quality via user feedback signal.
Guardrail false positives block legitimate requests
High
Medium
All guardrails default to flag-only; block mode requires explicit workspace opt-in. Provide guardrail override header for trusted internal services. Monitor false-positive rate in Grafana.
Hot-path Rust panic crashes pod
High
Low
Use catch_unwind in request handlers. Tokio task panics are isolated. Kubernetes liveness probe restarts unhealthy pods. Circuit breaker prevents traffic to crashed pods.

13. Glossary
Term
Definition
ANN
Approximate Nearest Neighbour — algorithm used by Qdrant to find semantically similar prompt embeddings
Circuit Breaker
Pattern that stops routing requests to a failing provider and re-tests periodically for recovery
EWMA
Exponentially Weighted Moving Average — used to smooth observed latency measurements for routing decisions
FR
Functional Requirement — a specific capability the system must provide
Guardrail
A middleware stage that scans or filters requests/responses against a policy
NFR
Non-Functional Requirement — a quality attribute (performance, availability, security)
ONNX Runtime
Open Neural Network Exchange Runtime — used to run embedding models in-process without Python overhead
OTel
OpenTelemetry — vendor-neutral observability standard for traces, metrics, and logs
PII
Personally Identifiable Information — any data that can identify a natural person
Semantic Cache
Cache that matches requests by meaning (embedding similarity) rather than exact string match
SSE
Server-Sent Events — HTTP streaming protocol used to forward LLM token streams to clients
TPM
Tokens Per Minute — a rate-limit dimension used by LLM providers and enforced by the gateway
TTFT
Time-To-First-Token — latency from request submission to receipt of the first completion token; key UX metric for streaming
Virtual Key
Gateway-issued API credential (lgw_*) that maps to a real provider credential, enabling policy enforcement without exposing secrets
vLLM
A high-throughput inference engine for self-hosted LLMs compatible with the OpenAI API
